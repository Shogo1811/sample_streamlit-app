# お客様提出用 仕様書生成

あなたは仕様書作成の専門家です。現在の実装コードを正（Single Source of Truth）として、お客様提出用の仕様書を生成してください。

## 前提条件
- 仕様書はコードの実装を正として作成する（設計書ではなく、実装の正確な反映）
- 開発過程で複数のAIDLCサイクルを経て仕様が変遷している可能性があるため、常に最新の実装から生成する

## 実行手順

### Step 1: 既存仕様書の保全

1. `docs/specifications/` ディレクトリに既存の仕様書があるか確認する
2. **既存仕様書がある場合**:
   - `docs/specifications/archive/{YYYYMMDD}/` ディレクトリを作成
   - 既存の仕様書をすべてアーカイブディレクトリにコピー（移動ではなくコピー）
   - コピー完了後、元の仕様書を削除
   - `docs/specifications/archive/{YYYYMMDD}/README.md` に以下を記録:
     ```markdown
     # Archived Specifications
     **Archive Date**: {YYYY-MM-DD HH:mm:ss}
     **Reason**: New specification generated from current implementation
     **Files**:
     - {アーカイブしたファイル一覧}
     ```
3. **既存仕様書がない場合**: Step 2 に進む

### Step 2: コードベース分析

1. ワークスペース全体のソースコードを読み取る
2. 以下を分析・整理する:
   - アプリケーションの全体構成
   - 各モジュール/コンポーネントの機能
   - データモデル・データフロー
   - API・インターフェース仕様
   - 外部連携（Snowflake等）
   - 設定・環境変数
   - エラーハンドリング

### Step 3: 仕様書生成

`docs/specifications/specification.md` に以下の構成で仕様書を作成する:

```markdown
# {プロジェクト名} システム仕様書

**バージョン**: {バージョン番号}
**作成日**: {YYYY-MM-DD}
**ステータス**: レビュー待ち

---

## 1. システム概要
### 1.1 目的
### 1.2 対象ユーザー
### 1.3 システム構成図（ASCIIダイアグラム）

## 2. 機能仕様
### 2.1 機能一覧
### 2.2 各機能の詳細
（機能ごとに: 概要、入力、処理、出力、制約事項）

## 3. データ仕様
### 3.1 データモデル
### 3.2 データフロー
### 3.3 外部データソース連携

## 4. インターフェース仕様
### 4.1 画面一覧・画面遷移
### 4.2 各画面の仕様（レイアウト、項目、バリデーション）

## 5. 非機能要件
### 5.1 性能要件
### 5.2 セキュリティ要件
### 5.3 可用性要件

## 6. 外部連携仕様
### 6.1 Snowflake連携
### 6.2 その他の外部サービス

## 7. 設定・環境
### 7.1 環境変数一覧
### 7.2 設定ファイル

## 8. 制約事項・前提条件

## 9. 用語集
```

### Step 4: 専門家レビュー

仕様書の生成が完了したら、`/project:review` コマンドと同等のレビューを実施する。
具体的には、以下の専門家サブエージェントを **並列で** 起動し、仕様書の品質をレビューする:

1. **Backend Expert** — 技術的正確性、実装との整合性
2. **Frontend Expert** — UI仕様の完全性、ユーザビリティ記述
3. **QA Expert** — テスト可能性、曖昧さの検出
4. **Legal Expert** — 個人情報・コンプライアンス記述の適切性
5. **Security Expert** — セキュリティ要件の網羅性

各専門家は100点満点でスコアリングし、90点以上を合格とする。
不合格の場合は指摘事項を修正し、再レビューを行う。

### Step 5: 完了メッセージ

```markdown
> **📄 仕様書生成完了**
> 仕様書: `docs/specifications/specification.md`
> {アーカイブした場合: 旧版アーカイブ: `docs/specifications/archive/{YYYYMMDD}/`}
>
> **レビュー結果**: {PASS or FAIL}
> {結果の要約}
>
> 人間による最終確認をお願いします。
```

## 重要なルール
- 実装コードを正とする（設計書や既存仕様書ではなく、実際のコードから生成）
- 既存仕様書は必ず日付フォルダにアーカイブしてから新規作成する
- AIが古い仕様書を再度読み込んで混乱しないよう、アーカイブは別領域に保全する
- 仕様書にはASCIIダイアグラムを使用する（画像ファイルは使わない）
- 専門家レビューで90点未満の項目があれば修正してから完了とする
