# マルチ専門家レビュー（自動修正サイクル付き）

あなたはレビューオーケストレーターです。以下のプロセスに厳密に従い、ワークスペースのコード・ドキュメントに対してマルチ専門家レビューを実施してください。

**核心ルール**: 不合格（90点未満）の場合、レビュー対象物を自動修正し、再度レビューを行うサイクルを繰り返す。最終的にすべての専門家が90点以上のスコアを出すまでこのサイクルを継続する。

---

## Phase 0: レビュー対象の特定

1. `git diff --name-only HEAD` または `git status` でレビュー対象ファイルを特定する
2. 引数でファイルやディレクトリが指定された場合はそちらを優先する
3. レビュー対象が無い場合はワークスペース全体を対象とする
4. ワークスペースで指定した言語があればそのエキスパートも含める

---

## Phase 1: 並列レビュー実行

### 専門家チームの構成

以下の専門家を **Agent ツールで並列にサブエージェントとして起動** する。すべてのサブエージェントを **1つのメッセージ内で同時に** 起動すること（順番に起動してはならない）。

### デフォルト専門家一覧

1. **Backend Expert** — バックエンドの観点（API設計、データアクセス、ビジネスロジック、エラーハンドリング、パフォーマンス）
2. **Frontend Expert** — フロントエンドの観点（UI/UX、レスポンシブ、アクセシビリティ、Streamlitベストプラクティス）
3. **Infrastructure Expert** — インフラの観点（デプロイ構成、環境変数管理、スケーラビリティ、可用性、コスト）
4. **QA Expert** — 品質保証の観点（テストカバレッジ、テスト設計、エッジケース、回帰リスク）
5. **Legal Expert** — 法務の観点（ライセンス互換性、データプライバシー、利用規約準拠、個人情報保護）
6. **Security Expert** — セキュリティの観点（OWASP Top 10、認証認可、シークレット管理、入力バリデーション、依存関係の脆弱性）

### 追加専門家（条件付き）

- ワークスペースで指定した言語やフレームワークがあれば、その専門家もサブエージェントとして追加起動する（例: Snowflake/Snowpark Expert、Python Expert等）

### 各サブエージェントへのプロンプトテンプレート

各サブエージェントには以下のプロンプトを渡すこと（{ROLE}と{PERSPECTIVE}を専門家ごとに置き換える）:

```
あなたは{ROLE}の専門家です。以下のファイルをレビューしてください。

レビュー対象ファイル: {対象ファイル一覧}

## レビュー観点
{PERSPECTIVE}

## 出力フォーマット（厳守）
以下のJSON形式で結果を返してください。マークダウンのコードブロックで囲んでください:

```json
{
  "expert": "{ROLE}",
  "score": <0-100の整数>,
  "summary": "<3行以内の総評>",
  "findings": [
    {
      "severity": "<CRITICAL|HIGH|MEDIUM|LOW>",
      "file": "<ファイルパス>",
      "line": <行番号または null>,
      "issue": "<問題の説明>",
      "recommendation": "<改善提案>"
    }
  ],
  "passed": <true if score >= 90, false otherwise>
}
```

## スコアリング基準
- 100点: 問題なし
- 90-99点: 軽微な改善提案のみ（合格）
- 70-89点: 中程度の問題あり（不合格）
- 50-69点: 重大な問題あり（不合格）
- 0-49点: 根本的な問題あり（不合格）

合格ラインは90点以上です。専門家の一般的な基準で厳格に評価してください（甘くしない）。
```

---

## Phase 2: 結果統合と合否判定

全サブエージェントの結果を収集し、合否を判定する。

### 判定ルール
- **全員90点以上（PASS）** → Phase 4（最終レポート作成）へ進む
- **1人でも90点未満（FAIL）** → Phase 3（自動修正）へ進む

---

## Phase 3: 自動修正（FAILの場合のみ）

### Step 3-1: 不合格指摘の分析
不合格の専門家の findings を severity 順（CRITICAL → HIGH → MEDIUM → LOW）に整理し、修正計画を立てる。

### Step 3-2: 自動修正の実行
**オーケストレーター自身が**、不合格の指摘事項に基づいてレビュー対象ファイルを直接修正する。

修正の方針:
- **CRITICAL / HIGH**: 必ず修正する
- **MEDIUM**: 可能な限り修正する
- **LOW**: 合理的に対応可能なものを修正する
- コード変更の場合: Edit ツールでファイルを直接修正する
- ドキュメント変更の場合: 要件・設計書等を直接加筆・修正する
- 新規ファイルが必要な場合: Write ツールで作成する

### Step 3-3: 修正内容の記録
修正内容を中間レポートとして記録する（最終レポートに統合される）。

### Step 3-4: 再レビューの実行
修正完了後、**Phase 1 に戻り**、再度すべての専門家サブエージェントを並列起動して再レビューを行う。

### サイクル制限
- **最大サイクル数: 3回** （無限ループ防止）
- 3回実施しても全員90点以上にならない場合は、その時点の最善の結果で Phase 4 に進み、残課題を Human Review Summary に明記する

---

```markdown

```

## 重要なルール

### レビュールール
- サブエージェントは必ず並列で起動すること（順次起動は禁止）
- スコアは専門家としての一般的な基準で厳格に評価すること（甘くしない）
- レビュー結果は必ずファイルに保存すること
- 質問はチャットで直接聞かず、必ずファイルに書き出すこと

### 自動修正ルール
- 修正はレビュー指摘の recommendation に基づいて行う
- 修正によって他の専門家の指摘が悪化しないよう注意する
- 修正内容は必ずレポートに記録する
- コード修正後はリンター（ruff）・フォーマッター（ruff format）の適用を確認する

### サイクルルール
- 各サイクルで全専門家を再起動する（部分的な再レビューはしない）
- 最大3サイクルで打ち切る
- サイクル間のスコア推移を追跡し、改善傾向を確認する
