# 要件定義書

## 意図分析サマリ

| 項目 | 内容 |
|------|------|
| **ユーザーリクエスト** | ラーメンチェーン店の食材等の自動発注を目的とした物流システムStreamlitアプリ（Snowflake連携） |
| **リクエスト種別** | 新規プロジェクト |
| **スコープ見積** | 複数コンポーネント |
| **複雑度見積** | 中程度 |
| **デプロイ先** | Streamlit in Snowflake (SiS) |

---

## 1. 機能要件

### 1.1 可視化ダッシュボード（初期リリース）

| ID | Requirement |
|----|-------------|
| FR-01 | 店舗別の在庫状況をリアルタイムで表示する（グラフ・チャート） |
| FR-02 | 食材カテゴリ別の在庫推移を時系列グラフで表示する |
| FR-03 | 配送状況（未配送・配送中・配送完了）を一覧およびステータス別に表示する |
| FR-04 | 店舗・期間・食材カテゴリでデータをフィルタリングできる |
| FR-05 | 在庫が閾値を下回っている食材をアラート表示する |

### 1.2 発注提案機能（初期リリース）

| ID | Requirement |
|----|-------------|
| FR-06 | 在庫データと消費パターンに基づき、発注が必要な食材の提案リストを表示する |
| FR-07 | 提案リストには食材名、現在庫数、推奨発注数、発注理由を含める |
| FR-08 | 店長が提案を確認し、承認/却下/数量修正を行える |
| FR-09 | スケジュールベース（定期発注）の提案も自動生成する（毎日/毎週の固定発注パターン） |
| FR-10 | 承認済みの発注提案を発注予定リストとして保存する（実際の発注実行は将来フェーズ） |

### 1.3 配送管理機能（初期リリース）

| ID | Requirement |
|----|-------------|
| FR-11 | ドライバーが自分の担当配送先リストとスケジュールを確認できる |
| FR-12 | ドライバーが配送完了の報告（ステータス更新）を行える（ボタン操作のみ） |
| FR-13 | 配送完了時に配送日時が自動記録される |
| FR-14 | 店長が自店舗への配送状況を確認できる |

### 1.4 自動発注実行（将来フェーズ）

| ID | Requirement |
|----|-------------|
| FR-15 | 承認済みの発注提案を外部システムに自動送信する |
| FR-16 | AI予測ベースの需要予測と自動発注ロジックを追加する |

### 1.5 認証・ロール管理

| ID | Requirement |
|----|-------------|
| FR-17 | Snowflakeのロール機能で「店長ロール」「ドライバーロール」の2種類のアクセス権限を分離する |
| FR-18 | 店長ロール: ダッシュボード閲覧、発注提案の確認・承認、配送状況確認 |
| FR-19 | ドライバーロール: 担当配送先の確認、配送完了報告（ボタン操作のみ） |

## 2. 非機能要件

### 2.7 セキュリティ

| ID | Requirement |
|----|-------------|
| NFR-08 | SQLインジェクション対策: 読み取りはSnowpark DataFrame API使用を必須。書き込みはStored Procedure（CALL文）経由に統一。session.sql()の使用はDDL/管理系コマンドに限定し、ユーザー入力値の埋め込みにはsnowflake.snowpark.functions.lit()を使用。文字列結合によるSQL構築を禁止 |
| NFR-09 | 入力バリデーション: サーバーサイドで必ず実施。発注数量は1以上10,000以下の整数。配送ステータスはホワイトリスト方式（「未配送」「配送中」「配送完了」のみ許可） |
| NFR-10 | 行レベルセキュリティ: Snowflake Row Access Policyを使用。適用マトリクス: (1) 在庫テーブル→店舗IDベースRAP、(2) 発注提案テーブル→店舗IDベースRAP、(3) 配送テーブル→ドライバーID+店舗IDベースRAP。RAP擬似ロジック: `SELECT related_id FROM user_role_mapping WHERE user_id = CURRENT_USER() AND role_type = '対象ロール'`で対象IDを取得し、テーブルの店舗IDまたはドライバーIDと照合する。APP_ADMIN_ROLEに対してもRAPはバイパスされない（EXEMPTロールに含めない）。SiS環境でCURRENT_USER()がログインユーザー名を返すことをCONSTRUCTION最初期に検証する |
| NFR-10a | ユーザーロールマッピングテーブルのセキュリティ: テーブルオーナーシップはSECURITYADMIN。APP_ADMIN_ROLEにはSELECTのみ許可。マッピングデータの変更はSECURITYADMIN経由の運用手順とし、変更履歴をACCESS_HISTORYで追跡可能にする |
| NFR-11 | 監査ログ: 発注提案の承認/却下/修正、配送ステータス更新の全操作を監査ログテーブルに記録する（操作者、日時、操作内容、変更前後の値）。テーブルオーナーシップはSYSADMIN。APP_ADMIN_ROLEにはINSERT ONLYのGRANTを設定し、UPDATE/DELETEは一切付与しない（APPEND-ONLY保証）。Snowflake ACCESS_HISTORY / QUERY_HISTORYも活用する |
| NFR-12 | 通信暗号化: SiS環境はHTTPS経由（Snowflake標準）。保存データ暗号化: Snowflakeデフォルト（AES-256）に依存 |
| NFR-13 | セッション管理: SiS環境のセッション管理はSnowflake側に委譲。アプリレベルでのセッションタイムアウトはStreamlitのデフォルト設定に準拠 |

## 3. 技術的意思決定

### 3.1 技術スタック

| コンポーネント | 技術 | 選定理由 |
|-----------|------------|-----------|
| Frontend | Streamlit | SiS環境での標準UI |
| Backend | Python (Snowpark) | SiS環境ではSnowpark Pythonがネイティブに利用可能 |
| Database | Snowflake | ユーザー指定。統合物流データを格納 |
| Deployment | Streamlit in Snowflake (SiS) | ユーザー指定。Snowflake内でアプリを直接デプロイ |
| Authentication | Snowflake Roles | SiS環境標準。店長ロール・ドライバーロールで分離 |
| Linter/Formatter | ruff | Pre-pushチェックで使用 |
| Testing | pytest | Pre-pushチェックで使用 |

### 3.2 Snowflakeコネクタ
SiS環境ではSnowpark Pythonがネイティブに利用可能なため、外部コネクタ（snowflake-connector-python等）は不要。`snowflake.snowpark` を使用してSnowflakeデータにアクセスする。

### 3.3 SiS環境固有の制約

| 制約 | 詳細 | 対応方針 |
|------|------|----------|
| パッケージ制限 | SiSではAnacondaチャネル（Snowflake提供）のパッケージのみ利用可能。任意のpipパッケージは不可 | 使用ライブラリはSiS対応パッケージリストで事前確認する |
| セッション管理 | `snowflake.snowpark.context.get_active_session()` でセッション取得。通常のSession.builder不可 | アプリ起動時に1回取得し、st.cache_resourceでキャッシュする |
| バックグラウンド処理不可 | SiSアプリはユーザーアクセス時のみ実行。cron/バックグラウンドジョブ不可 | スケジュール処理はSnowflake Tasksで実行し、結果テーブルをアプリで表示する |
| 外部ネットワーク制限 | SiSから外部APIへの直接通信は不可（External Access Integration必要） | 初期リリースでは外部通信不要。将来フェーズ（FR-15）で対応 |
| ファイルシステム制限 | SiSではローカルファイルシステムへのアクセスが制限される | データの永続化はすべてSnowflakeテーブルに行う |

### 3.4 Snowpark データアクセスパターン

| パターン | 用途 | 方針 |
|----------|------|------|
| DataFrame API | 読み取りクエリ（ダッシュボード、一覧表示） | collect()の呼び出しを最小限にし、サーバーサイドでフィルタリング・集計を実施 |
| session.sql() | DDL/管理系コマンドのみ | ユーザー入力値の埋め込み禁止。書き込み操作にはStored Procedure（CALL文）を使用 |
| Stored Procedure | ビジネスロジック（発注提案算出）、ステータス更新、監査ログ記録 | OWNER'S RIGHTS（EXECUTE AS OWNER）で定義。Procedure内でCURRENT_USER()とユーザーロールマッピングテーブルを照合して操作権限を検証。APP_ADMIN_ROLEにもテーブルへの直接UPDATE権限は付与せず、Procedure経由のみとする。状態遷移バリデーション（逆遷移禁止）を実施し、違反時はRAISE例外を返す |
| Snowflake Tasks | スケジュール実行（定期発注提案生成、データ保持期間管理） | CRON式でスケジュール設定。専用Warehouseを使用しコスト分離。TASK_HISTORY()による実行結果監視。タスク失敗時のアラート通知。SiSアプリ側でタスク最終成功実行時刻を表示 |

### 3.5 Snowflake Warehouse設定方針

| 設定項目 | 値 | 理由 |
|----------|-----|------|
| アプリ用Warehouse サイズ | X-Small | 50店舗規模では十分。必要に応じてスケールアップ |
| Auto-Suspend | 60秒 | コスト最適化。短時間の無操作でサスペンド |
| Auto-Resume | 有効 | ユーザーアクセス時に自動復帰 |
| Tasks用Warehouse | X-Small（別Warehouse） | アプリ用とコスト分離するため専用Warehouseを使用 |

### 3.6 ロールベースアクセス制御（RBAC）設計

```
ACCOUNTADMIN
  └── SYSADMIN
        └── APP_ADMIN_ROLE（アプリ管理者 — SiSアプリのオーナーロール）
              ├── MANAGER_ROLE（店長ロール）
              │     - SELECT: 在庫、配送、発注提案（自店舗のみ - Row Access Policy）
              │     - INSERT/UPDATE: 発注提案承認テーブル
              └── DRIVER_ROLE（ドライバーロール）
                    - SELECT: 配送テーブル（自分の担当のみ - Row Access Policy）
                    - UPDATE: 配送ステータス（自分の担当のみ）
```

- SiSアプリはAPP_ADMIN_ROLEで実行
- ロール判定はCURRENT_USER()ベースで、ユーザーロールマッピングテーブルと照合する
- Row Access Policy: CURRENT_USER()ベースで、店舗マスタのmanager_user_idまたはドライバーテーブルのsnowflake_usernameと照合し、対象データのみアクセス可能とする

### 3.7 概念データモデル

```
[店舗マスタ] 1---* [在庫] *---1 [食材マスタ]
     |                              |
     |                              |
     1                              *
     |                              |
[発注提案] *---1 [食材マスタ]      [発注予定]
     |
     *
[配送] *---1 [ドライバー]
     |
     *---1 [店舗マスタ]

[ユーザーロールマッピング]（CURRENT_USER()とロール・関連IDの対応）
[監査ログ]（独立テーブル — 全操作を横断的に記録、APPEND-ONLY）
[同意記録]（ユーザー同意の証跡管理）
```

主要エンティティ:
- **店舗マスタ**: 店舗ID、店舗名、住所、担当店長ユーザーID（Snowflakeユーザー名 — Row Access Policy照合用）
- **食材マスタ**: 食材ID、食材名、カテゴリ、単位、在庫閾値（システム固定値）
- **在庫**: 店舗ID、食材ID、現在庫数、最終更新日時
- **発注提案**: 提案ID、店舗ID、食材ID、推奨発注数、発注理由、ステータス（生成→確認中→承認/却下。逆遷移不可）、作成日時
- **発注予定**: 予定ID、提案ID、承認店長、承認日時、発注数量
- **配送**: 配送ID、ドライバーID、店舗ID、ステータス（未配送→配送中→配送完了。逆遷移不可）、予定日時、完了日時
- **ドライバー**: ドライバーID、氏名、Snowflakeユーザー名（Row Access Policy照合用）
- **監査ログ**: ログID、操作者、操作日時、操作種別、対象テーブル名、対象レコードID、変更前値、変更後値（APPEND-ONLY: INSERT権限のみ、UPDATE/DELETE禁止）
- **同意記録**: 記録ID、ユーザーID、同意日時、ポリシーバージョン
- **ユーザーロールマッピング**: ユーザーID（Snowflakeユーザー名）、ロール種別（MANAGER/DRIVER）、関連ID（店舗IDまたはドライバーID）

### 3.8 デプロイメント方針

| 項目 | 方針 |
|------|------|
| デプロイツール | Snowflake CLI (`snow streamlit deploy`) またはSnowsight UIでのGitリポジトリ連携。PROD環境への手動アップロードは禁止（CI/CD経由のみ） |
| 環境分離 | DEV_DB / STG_DB / PROD_DB のデータベース分離。環境別パラメータ: Time Travel（DEV:1日, STG:3日, PROD:7日）、Auto-Suspend（DEV:60秒, PROD:180秒） |
| CI/CD | GitHub Actionsで自動テスト・デプロイ。CONSTRUCTIONフェーズ初期にワークフローYAML作成。PRODデプロイはPRレビュー+STGテスト通過が必須 |
| CONSTRUCTION前提条件 | pyproject.toml（依存管理）、Snowflakeインフラ設定SQLスクリプト（setup.sql: Warehouse/Role/RAP/Resource Monitor）、GitHub Actionsワークフロー定義をCONSTRUCTION Code Generation Part 1で作成する |

### 3.9 利用規約・プライバシーポリシー

- 初回利用時にプライバシーポリシー同意画面を表示（st.checkboxによる同意取得）
- 利用規約・プライバシーポリシーのテキストはSnowflakeテーブルに格納し、更新可能にする
- 同意記録管理: 同意記録テーブル（ユーザーID、同意日時、同意対象ポリシーのバージョン）を作成し、証跡を保持する
- プライバシーポリシー改定時: 改定版を表示し、再度チェックボックスで同意取得。旧版同意のユーザーは再同意完了まで機能制限する
- 同意撤回フロー: (1) 撤回申請受付（システム外運用 — 管理者へ連絡）→ (2) アカウント機能停止（2営業日以内）→ (3) 個人データ匿名化（5営業日以内）。監査ログ内の個人情報は法的保持義務（NFR-14b対応）に基づき5年間保持後に匿名化する。取引記録（発注・配送数量等）は個人情報を含まない形で商法上の保存義務期間保持する
